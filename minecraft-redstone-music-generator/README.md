## 📖 项目概述

**Minecraft红石音乐投影生成器**是一款革命性的开源工具，它打破了音频与游戏世界之间的界限。通过将普通的音频文件（如MP3、WAV、OGG等）转换为Minecraft中的红石音乐装置，让任何热爱Minecraft的玩家都能轻松创作复杂的红石音乐，无需深入了解红石电路原理或音乐理论。

### 🎯 核心理念
本工具旨在**降低红石音乐创作门槛**，让每个Minecraft玩家都能：
- 将喜欢的歌曲转换为游戏内可播放的音乐
- 无需手动搭建数以千计的红石元件
- 专注于创作和分享，而非繁琐的技术实现
- 探索音频与游戏机制结合的新可能性

### 🌟 项目亮点
- **完全开源免费**：基于MIT许可证，代码透明可审计
- **跨平台支持**：Windows、macOS、Linux全平台运行

- **智能算法**：自动优化红石电路布局
- **多格式输出**：兼容主流Minecraft结构格式

## 🎵 详细使用教程

### 第一阶段：音频准备与上传

#### 选择合适的音频文件
**最佳实践**：
- **时长**：1-3分钟效果最佳（最长支持30分钟）
- **格式**：MP3 > WAV > OGG > M4A
- **比特率**：192kbps以上，320kbps最佳
- **声道**：立体声或单声道均可
- **复杂度**：避免过于嘈杂的背景音

**不推荐的音频特征**：
1. 人声过于突出的歌曲（乐器部分会被忽略）
2. 持续的低频噪音
3. 动态范围过大（极安静和极响亮的交替）
4. 版权保护的数字音乐（DRM保护）

#### 上传过程详解
1. **拖放上传**：
   - 将音频文件直接拖到"拖放音频文件到此处"区域
   - 区域会高亮显示，松开鼠标完成上传
   
2. **文件选择上传**：
   - 点击上传区域或"选择文件"按钮
   - 在弹出的文件对话框中选择音频
   
3. **上传后验证**：
   - 文件名、大小、时长信息自动显示
   - 简易波形图生成
   - 音频控件启用（播放、暂停、停止）

### 第二阶段：音频参数调整

#### 基础调整参数
| 参数 | 范围 | 默认值 | 作用 | 效果说明 |
|------|------|--------|------|----------|
| **音调调整** | -12 到 +12 半音 | 0 | 改变音高 | +12：升高一个八度，-12：降低一个八度 |
| **音高调整** | -2 到 +2 八度 | 0 | 整体升降调 | +1：整体升一个八度，适合低音音乐 |
| **播放速度** | 0.5x 到 2.0x | 1.0x | 改变速度 | 0.5x：慢速播放，2.0x：快速播放 |
| **音符密度** | 低/中/高 | 中 | 控制音符数量 | 高密度：更多音符，细节更丰富 |
| **最大音符数** | 50-2000 | 500 | 限制生成规模 | 2000：最多生成2000个音符盒 |

#### 高级设置解释
**自动调音**：
- **启用**：自动匹配到Minecraft音高（0-24）
- **禁用**：使用线性映射，可能产生非标准音高

**循环播放**：
- 在试听时启用，便于反复检查效果

**音频效果**（高级选项）：
- **回声**：模拟空间反射效果
- **混响**：添加环境氛围感
- **均衡器**：调整低音/中音/高音比例

### 第三阶段：试听与优化

#### 试听工作流程
1. **点击"试听调整后音频"**
2. **观察处理进度**：
   - 音频上传 → 参数调整 → 效果处理 → 生成预览
3. **播放控制**：
   - 使用播放器控件控制预览
   - 调整音量（右上角滑块）
   - 启用循环播放反复检查
4. **评估标准**：
   - 音调是否自然
   - 节奏是否准确
   - 是否有明显失真
   - 整体效果是否符合预期

#### 常见调整策略
**问题：音符过于密集**
- 解决方案：降低音符密度或减少最大音符数

**问题：音调不准确**
- 解决方案：微调音调参数，尝试±1半音

**问题：节奏感不强**
- 解决方案：提高速度（1.1x-1.3x）或调整密度

**问题：音频失真**
- 解决方案：关闭音频效果，使用原始音频

### 第四阶段：生成投影文件

#### 生成过程详解
点击"生成红石音乐投影"后，系统将执行以下步骤：

| 步骤 | 进度 | 耗时 | 说明 |
|------|------|------|------|
| **1. 文件上传** | 0-10% | 10-30秒 | 上传音频到服务器处理 |
| **2. 音频分析** | 10-30% | 15-60秒 | 提取节拍、音高、旋律特征 |
| **3. 音符提取** | 30-50% | 20-90秒 | 确定每个音符的时间、频率、音量 |
| **4. 红石映射** | 50-70% | 10-40秒 | 映射到Minecraft音符盒和红石电路 |
| **5. 电路优化** | 70-85% | 5-20秒 | 优化布局，减少冗余元件 |
| **6. 文件生成** | 85-95% | 5-15秒 | 创建.litematic或.schematic文件 |
| **7. 完成准备** | 95-100% | 1-5秒 | 文件准备就绪，可下载 |

#### 预估时间参考
基于音频时长和复杂度：
- **30秒音频**：约45-60秒处理时间
- **1分钟音频**：约1.5-2.5分钟
- **3分钟音频**：约3-5分钟
- **5分钟音频**：约5-8分钟

影响因素：
1. 服务器性能（CPU速度、内存大小）
2. 音频文件大小
3. 选择的参数复杂度
4. 当前系统负载

## ⚙️ 技术架构详解

### 音频处理流程
本工具采用多阶段处理管道：

```
原始音频 → 解码加载 → 特征提取 → 节拍检测 → 频率分析 → 音符映射 → 红石转换 → 结构生成
```

#### 1. 音频解码与加载
- 使用Librosa库支持多种格式
- 统一采样率至44100Hz
- 转换为单声道处理
- 标准化振幅范围（-1.0 到 1.0）

#### 2. 音乐特征提取
**节拍检测算法**：
- 基于频谱通量（spectral flux）
- 动态阈值适应不同音乐类型
- BPM范围：60-180（可检测）

**音高检测**：
- 快速傅里叶变换（FFT）分析
- 谐波乘积谱（HPS）改进精度
- 频率范围：50-2000Hz

**音量分析**：
- RMS（均方根）计算能量
- 动态范围压缩
- 归一化处理

#### 3. 红石映射算法
**音符映射规则**：
```
频率 (Hz)   →  Minecraft音高 (0-24)
20-2000     →  线性映射
音量 (dB)   →  红石信号强度 (1-15)
时间 (秒)   →  红石刻 (1秒 = 10刻)
```

**乐器选择逻辑**：
- 低频（<150Hz）：贝斯、低音鼓
- 中频（150-450Hz）：竖琴、吉他
- 高频（>450Hz）：钟、铁琴、钢琴

#### 4. 电路优化策略
**空间优化**：
- 紧凑排列音符盒
- 共享红石线路
- 垂直堆叠设计

**时序优化**：
- 对齐到红石刻边界
- 消除冗余延迟
- 并行触发机制

**元件优化**：
- 合并相邻音符盒
- 减少中继器数量
- 优化红石粉布局

## ⚠️ 限制与注意事项

### 技术限制

#### 1. 音频处理限制
**无法完美处理的情况**：
- **极端动态范围**：如交响乐中的极弱到极强
- **复杂和声**：四部以上和声可能丢失细节
- **人声主导**：人声频率与乐器重叠，难以分离
- **持续噪音**：背景杂音可能被误识别为音符

**精度限制**：
- 节拍检测精度：±10毫秒
- 音高检测精度：±5音分（半音的1/20）
- 时间映射精度：0.1秒（1红石刻）
- 音量映射：15级离散强度

#### 2. 红石机制限制
**Minecraft物理限制**：
- **最大距离**：红石信号传输15格衰减
- **时序精度**：最小时间单位0.1秒（1刻）
- **音符范围**：仅25个半音（0-24音高）
- **同时发声**：同一时刻音符盒数量有限

**游戏性能考虑**：
- 过多红石元件可能导致卡顿
- 大型结构需要足够内存
- 复杂电路可能产生延迟
- 区块加载影响音乐连续性

#### 3. 文件与性能限制
**文件大小限制**：
- 输入音频：最大100MB
- 输出结构：最大256×256×256方块
- 音符数量：最多2000个音符盒
- 处理时间：30分钟音频约需15分钟处理

**系统要求**：
- Python处理需要足够内存
- 大文件需要磁盘空间
- 实时预览需要Web Audio API支持
- 现代浏览器推荐

### 使用注意事项

#### 1. 音频准备建议
**最佳实践**：
1. **预处理音频**：
   - 使用Audacity等工具降噪
   - 标准化音量（-1.0dB峰值）
   - 转换为单声道简化处理
   - 裁剪到所需长度

2. **格式选择**：
   - WAV格式提供最佳质量
   - MP3 192kbps以上保证质量
   - 避免有损压缩多次转码

3. **内容优化**：

   - 避免人声为主的歌曲
   - 电子音乐、古典乐器效果最佳
   - 节奏明确的音乐更易识别

#### 3. 故障排除指南

**常见问题与解决**：

| 问题现象 | 可能原因 | 解决方案 |
|----------|----------|----------|
| **无法连接到后端** | 端口占用/防火墙 | 检查5000端口，关闭防火墙测试 |
| **音频上传失败** | 文件过大/格式不支持 | 压缩音频，转换格式，检查大小 |
| **处理时间过长** | 音频太长/系统负载 | 缩短音频，关闭其他程序 |
| **生成文件无法加载** | 格式不兼容/Mod问题 | 检查Mod版本，尝试不同格式 |
| **音乐效果不佳** | 参数不合适/音频问题 | 调整参数，预处理音频 |
| **内存不足错误** | 大文件处理/系统内存小 | 增加虚拟内存，使用64位Python |

**详细诊断步骤**：

**问题：生成的文件Mod无法加载**
```
排查流程：
1. 检查文件大小 → 正常（>1KB）
2. 检查文件扩展名 → 正确（.litematic）
3. 检查Mod版本 → 匹配游戏版本
4. 检查文件位置 → .minecraft/schematics/
5. 尝试其他音频 → 确定是否特定文件问题
6. 查看游戏日志 → 寻找具体错误信息
7. 测试其他结构 → 确认Mod正常工作
```

**问题：处理过程卡住或失败**
```
诊断步骤：
1. 查看浏览器控制台（F12） → 网络错误/响应
2. 查看后端终端输出 → Python异常信息
3. 检查系统资源使用 → 内存/CPU是否耗尽
4. 尝试简化参数 → 降低复杂度测试
5. 重启应用程序 → 清除临时状态
6. 检查依赖包 → pip list验证版本
```

#### 4. 性能优化建议
**处理速度优化**：
1. **缩短音频时长**：1-3分钟最佳
2. **降低采样率**：22050Hz可接受
3. **减少音符密度**：适当稀疏化
4. **关闭高级效果**：简化处理流程
5. **使用SSD存储**：加快文件读写

**内存使用优化**：
1. **分块处理大文件**：避免一次性加载
2. **及时清理临时文件**：自动清理机制
3. **限制并发处理**：单任务模式
4. **增加虚拟内存**：系统级别优化
5. **使用64位Python**：支持更大内存

**输出质量平衡**：
```
质量 vs 性能 权衡：
最高质量：WAV + 高密度 + 所有效果 → 处理慢，文件大
平衡点：MP3 192k + 中密度 + 基础效果 → 推荐设置
快速处理：MP3 128k + 低密度 + 无效果 → 速度优先
```
